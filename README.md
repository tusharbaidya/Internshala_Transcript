# Fellow Transcript Analyzer

A MERN stack web application that uses Ollama (local LLM) to analyze supervisor transcripts and assess DeepThought Fellows against a structured rubric.

---

## Quick Setup

### Prerequisites
- Node.js 18+
- MongoDB (local or Atlas)
- Ollama installed from [ollama.com](https://ollama.com)

### 1. Start Ollama

```bash
# Install and pull the model (one-time)
ollama pull llama3.2

# Ollama starts automatically. If not:
ollama serve
```

### 2. Backend

```bash
cd backend
npm install
# Edit .env if needed (defaults work for local Ollama + local MongoDB)
npm run dev
# → Server running on http://localhost:5000
```

### 3. Frontend

```bash
cd frontend
npm install
npm run dev
# → App running on http://localhost:5173
```

Open **http://localhost:5173** in your browser.

---

## Model Choice: `llama3.2` (3B)

I chose llama3.2 because:
- Runs on most laptops (8 GB RAM minimum)
- Good instruction-following for structured JSON output
- Fast enough for a usable demo (~20–60s per transcript)
- `mistral` is a solid alternative if llama3.2 is unavailable

Change the model in `backend/.env`:
```
OLLAMA_MODEL=mistral
```

## Architecture

```
Browser (React + Tailwind + Vite)
    ↓ POST /api/analysis/run
Express Server (Node.js)
    ↓ Builds structured prompt
    ↓ POST http://localhost:11434/api/generate
Ollama (llama3.2, running locally)
    ↓ Returns raw text (JSON)
Express parses + validates → saves to MongoDB
    ↓ Returns structured JSON
React renders tabs: Evidence | KPI | Gaps | Questions
```

- **Frontend**: React 18, React Router, Tailwind CSS, Vite
- **Backend**: Express.js (ESM), Mongoose
- **Database**: MongoDB (analyses are saved for history view)
- **LLM**: Ollama local API at `localhost:11434`

## Design Challenges Tackled

### Challenge 2: Structured Output Reliability
LLMs don't always return clean JSON. My approach uses a 3-tier fallback parser in `backend/utils/ollama.js`:
1. Direct `JSON.parse()` on trimmed output
2. Strip markdown fences (```json ... ```) then parse
3. Find the first `{` to last `}` and parse that slice

If all attempts fail, the request retries once. This handles the most common failure modes without complicating the happy path.

I also set `temperature: 0.2` (low) to make outputs more deterministic and consistent across runs.

### Challenge 4: Showing Uncertainty
The UI explicitly frames AI output as a **draft**:
- A yellow warning banner at the top of every result: *"This analysis is generated by AI... use it as a starting point, not a final verdict."*
- The footer reminds: *"⚠ AI output is a draft — always apply your own judgment."*
- The score is labeled with a rubric level (e.g., "Exceeding Expectations") — not just a number — so the intern understands *what* 7 means.

### Challenge 5: Gap Detection
The prompt explicitly lists all 8 assessment dimensions and instructs the model to identify which ones were NOT covered. Rather than trying to detect absence through heuristics, I ask the model to reason about absence directly — then display gaps in a dedicated red-tinted tab so they're visually distinct from positive findings.

## What I'd Improve With More Time

1. **Side-by-side transcript + analysis view** — The intern should be able to read the original transcript while reviewing evidence, with quotes highlighted inline.
2. **Streaming response** — Ollama supports streaming. Showing the analysis build in real-time would remove the "black box" 30-second wait.
3. **Multi-prompt architecture** — Run separate focused prompts for scoring, evidence, and gap detection. More reliable than one giant prompt, at the cost of 3× latency.
4. **Evidence confidence scores** — Ask the model to rate how confident it is for each extracted quote, so weak inferences are visually de-emphasized.
5. **Export to PDF** — Interns need to share this with their program managers. A one-click PDF export would close the loop.
6. **Prompt version tracking** — Store which prompt version produced each analysis so improvements can be compared historically.
